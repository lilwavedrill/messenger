## 1. Предметная область

Мессенджеры — класс приложений, обеспечивающих обмен текстовыми сообщениями
между пользователями практически в реальном времени. Основные функции такого
сервиса: аутентификация пользователей, ведение личных и групповых переписок,
доставка сообщений с минимальной задержкой (менее 200 мс с точки зрения
пользователя), хранение истории сообщений и поиск по ней.

В рамках практической работы разработан учебный мессенджер, реализующий
базовый набор функциональности (MVP): регистрацию и вход, создание чатов,
обмен сообщениями через WebSocket, поиск и историю с пагинацией, а также
управление участниками групповых чатов.

---

## 2. Анализ требований

### 2.1 Функциональные требования

1. Регистрация и вход пользователей с выдачей токена доступа.
2. Список чатов — личные (1-на-1) и групповые (с названием).
3. Отправка и получение сообщений в режиме реального времени.
4. История сообщений с пагинацией «вверх» (подгрузка более старых сообщений).
5. Управление участниками группового чата — только администратором.
6. Поиск по сообщениям в рамках выбранного чата.

### 2.2 Нефункциональные требования

- **Отзывчивость:** доставка сообщения от отправителя к получателю < 200 мс.
- **Надёжность:** сообщения не должны теряться при обрыве соединения,
  повторные отправки не должны создавать дублей (идемпотентность).
- **Безопасность:** пароли хранятся только в виде хэша, эндпоинты защищены
  токенами, каждое действие пользователя логируется в таблицу аудита.
- **Порядок:** сообщения в рамках одного диалога поступают в правильном
  хронологическом порядке.

### 2.3 Ограничения

- Стек — Python + PostgreSQL (согласовано с преподавателем).
- Транспорт для реалтайма — WebSocket (один из вариантов, разрешённых ТЗ).
- Обязательно наличие ролей: не менее трёх (у нас — `reader`, `writer`, `owner`
  внутри чата).

## 4. Архитектура решения

Приложение построено по трёхзвенной схеме:

```
    +----------------+     WebSocket    +----------------+
    |    Клиент      | <==============> |    Сервер      |
    |  (HTML + JS)   |    REST/HTTP     |   (FastAPI)    |
    +----------------+ ---------------> +----------------+
                                                |
                                                v
                                        +----------------+
                                        |   PostgreSQL   |
                                        +----------------+
```

- **Клиент** — одностраничный HTML-документ без сборщика, отдаётся сервером
  как статика. Работает с REST-API (регистрация, история, поиск) и держит
  открытый WebSocket с сервером для реалтайма.
- **Сервер** — Python + FastAPI. Совмещает REST-эндпоинты и WebSocket в одном
  процессе. Внутри работает in-memory `Hub`, хранящий множество активных
  подключений по каждому чату и рассылающий события подписчикам.
- **База данных** — PostgreSQL. Хранит пользователей, чаты, участников,
  сообщения, статусы сообщений и журнал аудита.

## 5. Структура базы данных

ER-диаграмма приведена на рис. 5.1 (файл `docs/er.png`).

**Основные таблицы:**

- `users` — пользователи (id, username, password_hash, display_name).
- `chats` — чаты (id, type, title, created_by).
- `chat_members` — участники чата и их роли внутри чата
  (chat_id + user_id — составной PK; role ∈ {reader, writer, owner}).
- `messages` — сообщения (id, chat_id, sender_id, text, client_msg_id).
  Уникальный ключ `(chat_id, client_msg_id)` гарантирует идемпотентность.
- `message_status` — статусы доставки по каждому получателю
  (status ∈ {sent, delivered, read}).
- `audit_log` — журнал действий пользователей.

### 5.1 Ключевые индексы

- `(chat_id, id DESC)` на `messages` — пагинация «вверх» O(log n).
- GIN `to_tsvector('russian', text)` — полнотекстовый поиск.
- `(user_id)` на `chat_members` — быстрая выборка «мои чаты».

### 5.2 Примеры SQL-запросов

**Получить список моих чатов (JOIN):**
```sql
SELECT c.*
FROM chats c
JOIN chat_members cm ON cm.chat_id = c.id
WHERE cm.user_id = :user_id
ORDER BY c.created_at DESC;
```

**История «вверх» с пагинацией:**
```sql
SELECT * FROM messages
WHERE chat_id = :chat_id AND id < :before_id
ORDER BY id DESC
LIMIT 50;
```

**Количество сообщений в чате по авторам (GROUP BY + подзапрос):**
```sql
SELECT u.username, COUNT(*) AS cnt
FROM messages m
JOIN users u ON u.id = m.sender_id
WHERE m.chat_id = (
    SELECT id FROM chats WHERE title = 'General' LIMIT 1
)
GROUP BY u.username
ORDER BY cnt DESC;
```

## 6. Модель авторизации и роли доступа

### 6.1 Аутентификация

Пароль пользователя **никогда не хранится в открытом виде**. При регистрации
пароль пропускается через bcrypt с солью:

```
плохо:     password = "my_password_123"
хорошо:    password_hash = bcrypt("my_password_123", salt)
```

При успешной проверке пары login+password сервер выдаёт JWT-токен по алгоритму
HS256. Токен содержит идентификатор пользователя (`sub`) и время истечения
(`exp` — 24 часа по умолчанию). Клиент передаёт токен в заголовке
`Authorization: Bearer <token>` для всех защищённых эндпоинтов, а для
WebSocket — в query-параметре при установлении соединения.

### 6.2 Роли

В системе используются роли **внутри чата**:

| Роль | Чтение | Отправка | Управление участниками |
|------|:------:|:--------:|:----------------------:|
| reader | ✓ | — | — |
| writer | ✓ | ✓ | — |
| owner  | ✓ | ✓ | ✓ |

Проверка роли выполняется на каждом защищённом эндпоинте.

### 6.3 Аудит

Все значимые действия (регистрация, вход, отправка сообщения, создание чата,
добавление/удаление участника) записываются в таблицу `audit_log` с указанием
пользователя, типа действия и полезной нагрузки в формате JSONB.

## 7. API сервиса

| Метод | Путь | Назначение |
|-------|------|-----------|
| POST  | `/auth/register` | Регистрация пользователя |
| POST  | `/auth/login` | Вход, выдача JWT |
| GET   | `/chats` | Список чатов пользователя |

| DELETE| `/chats/{id}/members/{user_id}` | Удалить участника |

## 8. Реализация функциональных требований

Таблица соответствия ТЗ → результат.

| # | Требование | Где реализовано | Как проверить |
|---|-----------|-----------------|---------------|
| 1 | Регистрация и вход | `app/routes_auth.py` | `POST /auth/register`, `POST /auth/login` |
| 2 | Хранение паролей в хэше | `app/auth.py`, `hash_password()` | В таблице `users` в поле `password_hash` — bcrypt-строка |
| 3 | JWT-авторизация | `app/auth.py`, `current_user_id()` | Запрос без токена → 401 |
| 4 | Личные чаты | `app/routes_chats.py`, `type='direct'` | Создать чат `type=direct, member_ids=[X]` |
| 5 | Групповые чаты | `app/routes_chats.py`, `type='group'` | Создать чат с названием и списком участников |
| 6 | Роли в чате | `chat_members.role` | Добавить читателя, попытаться писать → 403 |
| 7 | Управление участниками (owner) | `POST/DELETE /chats/{id}/members` | Не-owner получает 403 |
| 8 | Реалтайм-обмен через WebSocket | `app/routes_ws.py`, `app/hub.py` | Открыть два клиента, отправить сообщение — второй получает push |
| 9 | История с пагинацией | `GET /chats/{id}/messages?before_id=` | Получить 50, потом дозагрузить более старые |
| 10 | Поиск по сообщениям | `GET /chats/{id}/search?q=` | Поиск по подстроке в тексте |
| 11 | Идемпотентность | `client_msg_id` + `UNIQUE` | Два одинаковых `POST` → в БД одна запись |
| 12 | Статусы (доп.) | `message_status`, WS `ack`/`read` | В UI: одна ✓ → две ✓✓ → две синих |
| 13 | Аудит действий | `app/models.py::AuditLog` | Все действия → в `audit_log` |

## 9. Инструкция по запуску

Требуется macOS/Linux, Python 3.11+, PostgreSQL 14+.

```bash
# 1) Клонировать репозиторий
git clone <url>
cd messenger

# 2) Поднять БД
psql -U postgres -c "CREATE DATABASE messenger;"
psql -U postgres -d messenger -f backend/schema.sql

# 3) Настроить окружение
cp backend/.env.example backend/.env
# отредактировать DATABASE_URL и JWT_SECRET

# 4) Установить зависимости и запустить сервер
cd backend
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
.venv/bin/uvicorn app.main:app --port 8000
```

После запуска:
- Веб-клиент: http://127.0.0.1:8000/app/
- Swagger UI: http://127.0.0.1:8000/docs
- Health-check: http://127.0.0.1:8000/healthz

## 10. Выводы

В ходе практической работы разработан рабочий MVP мессенджера, покрывающий
все обязательные функциональные требования ТЗ (кейс №1). В качестве
дополнительной функциональности реализованы статусы доставки сообщений
(sent / delivered / read) с уведомлениями в реальном времени.

Основные технические решения:
- Использование WebSocket в связке с REST — сообщения записываются
  транзакционно через REST-эндпоинт, а WebSocket служит каналом доставки.
  Такой подход упрощает обеспечение надёжности: даже при потере WebSocket
  сообщение уже сохранено в БД.
- Идемпотентность через `client_msg_id` защищает от дублей при ретраях.
- Пагинация по `id < before_id` вместо `OFFSET` — не создаёт «дыр» и дублей
  при появлении новых сообщений во время прокрутки.

Возможные направления развития: замена in-memory `Hub` на брокер сообщений
(например, Redis Streams) для масштабирования на несколько инстансов,
добавление вложений через объектное хранилище S3, шифрование сообщений на
стороне клиента.
